## tl;dr

The first tagged release of Clifford ships today. `cliffordc compile foo.cl`
produces an LLVM IR module that runs on bare-metal Cortex-M3 in QEMU. The
language surface is feature-complete for the v0.1 firmware target: pure
`@fn`s, mono- and multi-state `#automaton`s, `#effect` / `#interrupt` /
`#transition` callables, register-block MMIO with volatile loads/stores,
sigma loops, `if`/`else`, `let mut` accumulators, the full integer +
comparison + bitwise + shift surface, and Decision #22's
Acquire/Release/SeqCst memory-ordering fences.

`cliffordc` invokes the compiler from the command line, runs every
upstream semantic gate (sigil-layer + mutation-auth + totality +
categorical structure + mutation-profile validation + call-graph cycle
detection), and renders errors with `codespan-reporting`-style source
snippets. Tag: `v0.1.0`. Get it at <https://github.com/westerngazoo/Clifford>.

## What you can build with v0.1

Two end-to-end samples in the repo show off the full surface.

### 1. Dual-UART telemetry, ISR-driven, lock-free

```clifford
#automaton Uart1 {
  #address: 0x4000_4000;
  rx_data: u32 #offset: 0x00;
}

#automaton Telemetry {
  #states: [Idle, Active];
  bytes_uart1: u32;
  bytes_total: u32;
  last_byte:   u8;

  #transition tally_uart1 -> Active $ [Release] {
    Telemetry.bytes_uart1 += 1u32;
    Telemetry.bytes_total += 1u32;
    Telemetry.last_byte = (Uart1.rx_data as u8);
  }
}

#effect drain_total() -> u32 #mutates: [Telemetry] $ [Acquire] {
  return Telemetry.bytes_total;
}

#interrupt USART1_IRQ() #mutates: [Telemetry] #priority: HIGH {
  #> tally_uart1();
}
```

Two UART receive ISRs feed counters into a shared `Telemetry` automaton.
The `$ [Release]` on each tally transition pairs with the `$ [Acquire]`
on `drain_total` — the cross-context handoff is correct without any
locks, atomics, or critical sections. The compiler emits the LLVM IR
fences automatically; the source code never says `__atomic_thread_fence`.

That `$ [Release]` does three things in the lowering: writes the new
state-tag (`Active = 1`), emits a `fence release` LLVM instruction
(which becomes a `DMB` on Cortex-M), then `ret void`. The order is
guaranteed: the new state is published only after the fence makes
prior writes visible. This is the contract Decision #22 implies, and
slice 9 + slice 13 + slice 22-codegen-consumer make it real.

### 2. CRC-32, pure-functional, runs anywhere

```clifford
@fn crc32_byte(crc: u32, byte: u8) -> u32 {
  let mut c: u32 = crc ^ (byte as u32);
  sigma bit in 0u32..8u32 {
    if (c & 1u32) == 1u32 {
      c = (c >> 1u32) ^ 0xEDB88320u32;
    } else {
      c = c >> 1u32;
    }
  }
  return c;
}
```

Zero `#`-layer constructs. No automaton, no MMIO, no
target-specific anything. Compiles to the same LLVM IR that
`clang --target=x86_64-linux-gnu` or `clang --target=aarch64-darwin`
or `clang --target=riscv32imc-unknown-none-elf` happily consumes. The
canonical CRC-32 test vector (`"123456789"` → `0xCBF43926`) verifies
on every backend LLVM has.

This sample exists deliberately. Clifford's claim is "general-purpose
systems language whose unifying property is GA-verified concurrency
safety, with embedded firmware as the first canonical target." The
`#`-layer is for state machines + mutation; the `@`-layer is plain
functional code. CRC-32 is the smallest meaningful demonstration that
the `@`-layer alone is a complete language for hosted use.

## What `cliffordc` actually does

```bash
$ cliffordc compile examples/crc32.cl
cliffordc: wrote examples/crc32.ll (2510 bytes from examples/crc32.cl)

$ cliffordc compile examples/bad.cl
error[[parse]]: E0204: expected expression, found Semi at byte 51
  ┌─ examples/bad.cl:3:14
  │
3 │   return x + ;  // syntax error
  │              ^ here
```

Source-line diagnostics (slice 16) make errors actionable — line +
column, the offending source line, a caret pointing at the exact
byte. Phase prefixes (`[parse]`, `[resolve]`, `[check]`, `[effect]`,
`[codegen]`) tell you which gate caught the problem. The pipeline
runs every upstream check before codegen, so programs that violate
sigil-layer rules, mutation-auth, totality, or category structure
fail fast with a structured diagnostic.

The bare-metal proof: a CI workflow on every PR runs
`cliffordc compile tests/qemu/firmware_smoke.cl`, cross-compiles the
`.ll` with `clang --target=thumbv7m-none-eabi`, links against a
hand-written 90-line C startup + linker script targeting the
Stellaris LM3S6965 memory map, and runs the resulting ELF on
`qemu-system-arm -M lm3s6965evb` under ARM semihosting. Eight
functional checks across the v0.1 surface — `answer()`, `clamp(x)`,
`sum_to(n)`, `bit_count(x)` — verify the binary actually runs.

## Where the GA orthogonality engine fits in

`clifford-ortho` is the heart of the language's signature claim — the
crate that, given a program, proves "for every pair of computations
that the engine permits to run concurrently, their behaviour
multivectors wedge to a non-zero blade of the right grade." That's
the constructive existence proof for the product category C_X × C_Y.
The wedge-product lifts the question of "are these computations
disjoint?" to a single XOR-and-bitmask-comparison.

In v0.1 the crate exposes only the primitive — `outer_product(a, b) =
if a & b != 0 { None } else { Some(a | b) }`. That's the
algorithmically interesting line; everything else is the lifting
infrastructure (basis assignment, behaviour-multivector
construction, the `#mutates` set as a blade, race detection). The
v0.1 codegen happily compiles programs that the §7 verifier would
reject — there is no §7 verifier yet to invoke. v0.2's first slice
is wiring the verifier in.

That ordering is intentional. The compiler's existence is the
prerequisite for testing the verifier on real programs. Now that
v0.1 ships, v0.2 can build the verifier against a real
test corpus instead of synthetic toy inputs.

## Honest about what's deferred

Read the [v0.1.0 CHANGELOG entry](../../CHANGELOG.md#010--2026-05-08)
for the full deferred list. The headline omissions:

- **`clifford-ortho` §7 verifier** — the language's safety claim. v0.2
  slice 1.
- **Compound assign on locals** — `x += 1` works on automaton fields
  but not on `let mut` locals. Workaround: `x = x + 1`.
- **`break` / `continue`**, **`match`**, **methods**, **strings**,
  **`@snapshot`** — all v0.2+.

None of these block the v0.1 firmware target. They do mean that v0.1
is a *firmware-ready* compiler, not yet a *general-purpose ready*
compiler. That gap closes through v0.2.

## What v0.1 proves

- The architecture works. A 13-slice pipeline (lexer → parser → AST →
  resolve → types → check → effect → codegen) lowers a
  decisions-driven language to LLVM IR with zero ad-hoc passes.
- The sigil-layered surface (`@`-functional, `#`-imperative) reads
  cleanly. Two end-to-end samples — multi-producer telemetry and
  pure CRC-32 — exercise the full surface in ~150 source lines
  combined.
- The Decision #22 fence-ordering contract — write state, fence,
  ret — is the right shape for cross-context handoff without locks.
- Source-line diagnostics + phase-prefix routing make a 7-phase
  pipeline debuggable.

What v0.1 does NOT yet prove is the central thesis: that the GA
orthogonality engine catches concurrency hazards the type system
can't. That's v0.2. But it's now *testable*, which it wasn't before.
